GET-001: same sub project at two paths leaves a missing anchor

When a parent mounts the SAME sub project at two gitlink paths, the second mount's <wt>/<path>/.be anchor is not written, so the mount looks absent. The goal is that each gitlink path gets its own secondary-wt anchor into the one shared shard, however many paths reference the project. See Submodules, GET, Store. Repro: get/32-sub-same-project-twice (deterministic; "first/.be anchor missing").

Issues

Sub mount keys on the project shard, not the per-path mount, so the duplicate path is treated as already done.

  • SNIFFSubMount (sniff/SUBS.c) likely short-circuits on the shard's presence and skips re-seeding the per-path anchor.
  • Both mounts share one shard (one pool per project), but each PATH still needs its own .be sentinel; the second's is missing.

Blockers

None obvious; confirm where the per-path anchor write is gated out on the duplicate-shard path.

Planned

Write the per-path anchor independent of whether the shard already exists from a sibling mount.

  • In SNIFFSubMount, separate "shard exists" (skip fetch) from "anchor exists" (skip seed); always ensure the per-path .be.
  • Keep get/32 as the regression.

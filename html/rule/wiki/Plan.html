Plan: Beagle's live state in five fixed sections

Plan is the project's single live planning page and the template every tracking doc copies. The goal is one greppable place that answers what we want, what is wrong, what we do next, what stops us, and what is done — so no status lives only in code comments or a scratch todo. The method is five fixed sections (Goals, Issues, Action plan, Blockers, Landed) in WikiWeb shape, with each concrete discrepancy split out into a numbered DIS ticket under todo/ that repeats the same lifecycle sections (Issues, Blockers, Planned, Landed).

The five sections every plan doc carries

A plan doc is Goals (why), Issues (bugs, spec-vs-impl gaps, missing code), Action plan (ordered next steps), Blockers (what is waiting on a decision or upstream), and Landed (what shipped, newest first).

  • Goals: the durable intent — what the system should do — independent of any one work item.
  • Issues: every current discrepancy, one bullet each, linking its DIS ticket; three flavors — bug, codebase-contradicting-spec, missing code.
  • Action plan: the ordered steps in flight now; each step is land-and-stay-green, not a wishlist.
  • Blockers: anything that cannot proceed without a user ruling, an upstream fix, or another ticket landing first.
  • Landed: shipped work newest-first, kept as the regression memory so closed items are not re-opened by accident.

Discrepancies become numbered DIS tickets

A discrepancy found while testing (test surfaces a gap between spec and impl) is filed as todo/DIS-NNN.mkd, not patched silently into a test; the ticket owns the detail and Plan only links it.

  • One ticket per discrepancy: DIS-001, DIS-002, … each a WikiWeb page with Issues, Blockers, Planned, Landed sections.
  • Plan's Issues section is the index: a one-line claim per open ticket; the ticket carries the repro, root cause, and fix shape.
  • When a ticket fully lands, move its line from Issues to Landed here and mark the ticket's own Landed section.

Goals

Beagle is a git-compatible revision-control system with a uniform HTTP-verb/URI command language, syntax-aware diff/merge, and a multi-project ~/.be store built for parallel-worktree, rebase-centric workflows (see Verbs, Store).

  • Verb × URI shape is unambiguous and orthogonal; POST is the only commit-maker, PUT the only ref-writer (see Verbs).
  • Merges are token-level CRDT weaves with no false conflicts; every recorded commit is a tested commit (patch→test→post).
  • One store per machine, many worktrees; per-wt state in <wt>/.be/wtlog, branches/tags as flat reflog rows (see Store, Home).
  • The wiki (WikiWeb) is the spec; code is reconciled to it case by case, and gaps are filed as DIS tickets, not hidden.

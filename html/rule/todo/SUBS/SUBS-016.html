SUBS-016: latent O_TRUNC seed of a submodule's refs/wtlog (sniff/SUBS.c:410/420)

When seeding a submodule's trunk dir on (re-)fetch, sniff/SUBS.c creates the sub's refs (line 410) and wtlog (line 420) with FILECreate, which opens O_TRUNC (the code's own comments at :339/:394 acknowledge it). It is gated by if (!already_mounted) (line 402), where already_mounted = SNIFFSubIsMount(reporoot, path) (line 345) — so *today it only fires when the sub is not mounted, i.e. there is nothing to wipe, and it is safe. The defect is structural*: a single boolean stands between a live refs/wtlog and ftruncate(0). If SNIFFSubIsMount ever returns a false-negative (reports not-mounted while a live sub log exists), the seed truncates that live log to 0 — the exact [ULOG-001] zeroing class, in the submodule seed path. Harden by construction: never O_TRUNC a log path. See Issues, CLAUDE, [ULOG-001], Submodules.

Issues

A guarded-but-destructive seed of live-log paths.

  • sniff/SUBS.c:410 (refs) and :420 (wtlog) use FILECreate = O_TRUNC; the only guard is !already_mounted (line 402) via SNIFFSubIsMount (line 345).
  • No live-data loss observed — the guard holds today — but it is a latent O_TRUNC-on-live-log: defense-in-depth says a refs/wtlog open must never truncate, independent of any mount check.
  • Same failure class as [ULOG-001] (be zeroing a live log), just reached through the sub-seed path rather than ULOGClose.

Blockers

None. To repro the latent case you must force SNIFFSubIsMount to a false-negative with a live log present (a fault-injection/unit harness), since the production guard otherwise prevents it.

Planned

Seed create-if-missing, never truncate.

  • Repro-first (CLAUDE §17): a unit harness that calls the seed path with a non-empty refs/wtlog present and the mount check forced false, asserting the existing log is *preserved* (currently it would be zeroed).
  • Route the refs/wtlog seed through ULOGOpen (create-if-missing semantics, never O_TRUNC), or an explicit O_CREAT|O_EXCL create-only, so an existing log can never be wiped regardless of the already_mounted guard.
  • Keep the !already_mounted fast-path as an optimization, not as the safety boundary.

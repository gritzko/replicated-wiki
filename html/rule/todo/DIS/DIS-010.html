DIS-010: wtlog POST boundaries diverge from spec

POST/Sniff define two wtlog boundaries anchored at the latest GET: the pd boundary is the most recent get/post row, and the patch boundary is the most recent get or commit-all post. The impl computes the pd floor from the most recent post only (ignoring get) and resets the patch boundary at any post (no commit-all test), so a checkout or a selective commit silently changes what the next POST consumes. The goal is boundary scans that match the spec. See POST, Sniff.

Issues

Two distinct boundary bugs in sniff/AT.c, both reachable.

  • pd floor ignores get: SNIFFAtLastPostTs (AT.c:630-641) matches only the post verb, never get. Sequence put fileA → get ?branch → post leaves the floor at the pre-put post, so the stale pre-checkout put leaks into the new commit's scope and flips it into selective mode (Sniff says the intervening get resets pd).
  • patch boundary resets at any post: SNIFFAtPatchChain/PatchEntries (AT.c:558-564,593-597) start the scan at the most recent get or post, with no commit-all check. After a selective post, a following POST drops every patch row before it — foster/parent/picked headers are silently omitted and message resolution misses those rows. Commit-all detection (POST §"Boundaries", sniff/INDEX.md:59-63) is entirely absent.

Blockers

None; needs the commit-all predicate (no put/delete between a post's pd boundary and itself) implemented and wired into the patch scan, plus get added to the pd floor match.

Planned

Repro each (CLAUDE.md rule 17), then fix the two scans.

  • Repro A (pd): put f; get ?br; post must NOT carry the pre-checkout put.
  • Repro B (patch): patch ?X; put f; post; post — the second post must still record ?X's foster header.
  • Add get to the SNIFFAtLastPostTs floor; add a commit-all test so only commit-all post rows reset the patch boundary.

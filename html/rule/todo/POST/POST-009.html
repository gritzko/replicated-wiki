POST-009: be post be://host?/proj push — recv-side wt-advance recreates the peer's project shard → BEPRJDUP + be get ? exit 153

Pushing with the ?/<project> query form to a beagle peer half-works: objects transfer and the peer trunk ref FF-advances, but the receive side then tries to advance the peer worktree and *recreates the project shard in the peer's own HOME*, hitting BEPRJDUP ("project shard already exists at .be/<name>") and a be get ? exit 153, leaving the peer wt "at old tip … run be get ? manually". Split out of [POST-008] (the scheme-gate fix); this is a distinct recv/HOME-resolution defect. Surfaced both via be put be://localhost?/beagle (landing DIS-030+031 from a clone) and the POST-008 worker's ?/proj probe. The <path> form (be post file://<abs>) does NOT hit this — only the ?/<proj> selector. See POST, GET, CLAUDE.

MEM-039: js — io.mmap SIZE_MAX buffer-overrun on a path-conversion failure, plus an mmap ownership leak

js/io.cpp has two error-path memory bugs in the same FFI surface. (1) buffer-overrun: io.mmap(path) writes the path via JSStringGetUTF8CString then does u8sFeedn(u8bIdle(path), page, len-1) with no check that len>0 — if JSC returns 0 on a conversion error, len-1 wraps to SIZE_MAX and the feed runs a massive OOB. (2) leak-on-error: JABCioFileMap FILEMapRO's a live mmap meant to transfer to the JS GC via the typed-array deallocator, but if the JSObjectMakeTypedArray* creation FAILS the ownership transfer never happens and the mmap leaks. js is WIP (JSC-blocked), but both are real and on the error path. From the MEM sweep.

Issues

Two error-path defects in the JSC io bindings.

  • io.cpp:430 buffer-overrun: len-1 underflows to SIZE_MAX when JSStringGetUTF8CString returns 0, driving u8sFeedn OOB.
  • io.cpp:439-442 leak-on-error: the FILEMapRO mmap leaks when typed-array creation fails before ownership transfers to the GC deallocator.
  • Trigger 1: a non-UTF8-convertible / failing path arg. Trigger 2: a JSC typed-array allocation failure after a successful map.

Blockers

None blocking; js is JSC-WIP. Confined to io.cpp.

Planned

Guard the length and release the mmap on the creation-failure path.

  • Repro first (CLAUDE §17): a js test (or unit harness) feeding a path arg that makes JSStringGetUTF8CString return 0 → assert no OOB (ASAN); and a forced typed-array-create failure → assert the mmap is unmapped.
  • Add if (len == 0) <bail> before the len-1 feed.
  • On the typed-array-create failure branch, FILEUnMap the buffer before returning.

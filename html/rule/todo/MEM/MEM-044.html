MEM-044: spot — LSM open/merge leaks on error; SPOTOpenBranch hand-free + a logical double-unmap

spot's LSM plumbing acquires several mmaps/fds/heap tables then uses bare call()s that early-return without freeing them. SPOTOpenBranch (CAPO.c:1426) frees s->puppies + s->leaf_branch by hand on exactly two early error branches but leaks them on the others after acquiring s->puppies (kv64bAllocate :1407) and s->leaf_branch. CAPOMergeWorkers (CAPO.c:446) acquires wp (a kv64 puppy table holding every worker subdir's mmapped run) plus mbuf and leaks them on post-acquisition call() errors. Plus a benign-but-wrong logical double-unmap: capo_spot_replace (CAPO.c:692) does mapped=NULL on its LOCAL copy, so the caller capo_spot_file_cb (CAPO.c:855) FILEUnMaps the same booked descriptor again. From the MEM sweep.

Issues

LSM open/merge resources aren't uniformly released on error; one descriptor is unmapped twice.

  • CAPO.c:1426 SPOTOpenBranch: s->puppies (kv64bAllocate :1407) + s->leaf_branch leak on the error branches that don't hand-free them.
  • CAPO.c:446 CAPOMergeWorkers: wp (kv64 of per-worker mmaps) + mbuf leak on post-acquisition call() failures.
  • CAPO.c:692/855 double-unmap: capo_spot_replace clears only its local mapped, so the caller FILEUnMaps the same descriptor twice (benign via book, but wrong).

Blockers

None; spot-internal.

Planned

Uniform release of the LSM resources on every error; fix the local-NULL so the descriptor is unmapped once.

  • Repro first (CLAUDE §17): force an SPOTOpenBranch / CAPOMergeWorkers post-acquisition error; assert no map/fd/heap leak.
  • Route SPOTOpenBranch + CAPOMergeWorkers error exits through one cleanup that frees puppies/leaf_branch/wp/mbuf.
  • Make capo_spot_replace's unmap authoritative (clear the caller's pointer or don't unmap in the caller).

GIT-008: OFS writer — delta against delta bases, keep chains

Input

Context

The OFS-only writer never builds delta CHAINS: keep_find_raw_in_pack (keeper/KEEP.c:1642) accepts a base only when its stored record is RAW (type 1..4), skipping any base that is itself an OFS_DELTA. So when an object's natural base is itself a delta (the common case in a file's revision history — v₃'s base v₂ is stored as a delta), the writer cannot get the base bytes from a single inflate and stores the object RAW (a full blob, no delta). The format and reader already support OFS chains (keep_get_packed_rec subtracts ofs_delta and loops). This is purely a writer shortcut.

  • Measured (beagle.new, a fresh OFS-only clone, vs old ~/beagle): 54 MB vs 39 MB despite 2.7× FEWER objects (19043 vs 50873 — the clone GC'd ~31k unreachable objects); 6230 raw blobs vs 2156; ~2853 vs ~767 bytes/object. The lost chain compression more than offsets the GC win, so migration grew the store.
  • Compounded by [GIT-003] ingest re-deltifying (correctness floor, not deflate-passthrough): git hands beagle a well-chained pack on clone-back, then ingest re-stores it chain-broken.

Goals

  • Let the OFS writer delta against a base that is itself a delta, forming bounded-depth chains.
  • Bring a fresh OFS-only store BELOW the old REF-format size (recover chain compression).

Constraints

  • Resolve the delta base's bytes via the existing chain resolver (KEEPGet chases OFS/REF) — no re-roll; emit OFS_DELTA pointing at the base's offset in the open log.
  • Cap chain depth (git uses ~50) writer-side to bound read-resolution cost; keep the [MEM-022]-class hardening intact.
  • Base precedes dependent (forward-reference free); OFS bases stay within the one shard's open log (flat pool, [GIT-001]).
  • No pointer arithmetic, BASS/caller scratch not malloc, ABC style; do NOT edit abc/.

Outcome

  • Landed b01496da on dogs trunk: keep_find_raw_in_pack accepts OFS_DELTA bases (rejects REF); KEEPPackFeed resolves a delta base's bytes via keep_get_packed_rec and emits OFS chaining at its offset; new keep_chain_depth_at caps depth at KEEP_WRITE_CHAIN_MAX=50 (iterative, strictly-backward, MEM-022-safe — a base at the cap starts a fresh raw root).   • Re-measure (50876-object be: clone): bytes/object 2853 → 792, raw blobs 6230 → 2157, OFS-only (zero REF_DELTA), 10574 delta-of-delta objects (max depth 41). The GIT-008 regression is eliminated. Honest caveat: at equal object count the OFS-only store is 40.3 MB vs the old REF store's 39.3 MB (+2.5%); dipping below the old size needs [GIT-006] recompaction's better base selection, not clone-back.

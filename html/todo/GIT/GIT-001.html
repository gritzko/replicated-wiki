GIT-001: migrate keeper to a single-stream, OFS-only, offset-addressed pack log (umbrella)

Input

Context

Today a keeper log is many git packs back-to-back with a mixed delta scheme: OFS_DELTA pack-local, REF_DELTA resolved by hashlet anywhere in the pool (KEEPPackFeed picks REF whenever the base isn't raw in the same pack — keeper/KEEP.c:1724). That mix is the root of the store's hardest code: cross-file REF_DELTA recursion in the resolver (rdepth, KEEP_XFILE_RECUR_MAX, buf3 aliasing, the [MEM-022] stack-DoS hardening), the KEEPDIRTY "descendant references via REF_DELTA" branch-drop guard, and the bookmark-tiling wire serve whose count math caused [GET-019] truncation. This umbrella moves the store to ONE growing append-only pack per shard where every delta is OFS_DELTA against an earlier object *by offset only*; REF_DELTA / thin / arbitrary foreign packs are normalized away at the ingest boundary (the same quarantine philosophy as GITParseRef for refnames). See Store, PackLog, Keeper, CLAUDE.

Goals

  • Move the store to ONE growing append-only pack per shard, every delta OFS_DELTA against an earlier object by offset only.
  • Normalize REF_DELTA / thin / arbitrary foreign packs away at the ingest boundary so native logs and the resolver never see them.
  • Collapse the resolver's hardest code (cross-file REF_DELTA recursion, the descendant-REF branch-drop guard, bookmark-tiling serve math) by removing the mixed delta scheme.

Design decisions

The "object-by-object, not pack-by-pack" append (the chat ruling) commits us to the single-stream end state; the price is paid only at serve time, never in storage.

  • ONE stream per shard: a single growing NNNNN.keeper per project pool (per Store/PackLog: one flat object pool per project, branches as REFS rows — the impl already resolves all objects to <root>/.be/<project>/), no new-log-past-threshold rollover; objects appended individually at writer-assigned offsets.
  • OFS-only storage: native logs contain raw objects + OFS_DELTA exclusively; REF_DELTA never lands in a stored log, so a contiguous byte range is always self-resolvable and droppable shards keep no cross-file delta deps.
  • Boundary quarantine: foreign-format complexity (REF_DELTA, thin packs, OFS-in-foreign-pack) lives ONLY in UNPK (the foreign reader) and the GIT-003 ingest funnel; the native writer and resolver never see it.
  • Format core in dog/git, dogs as consumers: the native pack-log writer, the OFS-only offset resolver, and the OFS varint encoder are dog/git format functions (bytes + offsets, no refs/index/singleton), so keeper AND the js/JABC binding share one implementation. Storage policy (the one flat shard pool, refs, ingest, serve) stays in keeper; the binding stays pure marshalling — no pack-log logic in either consumer.
  • Offsets stay (file_id, offset): "offset-only" governs DELTA base references, NOT the LSM index — git's graph is sha-addressed (commit→tree/parents, tree entries, wire wants/haves), so the sha→(file,offset) index remains the entry boundary.
  • Serve-time cost only: full clone = blast the whole OFS-only log verbatim (it is a valid git pack sans checksum); incremental fetch needs a thin-pack builder (GIT-005), a pure presentation transform that never mutates storage.
  • Compression tradeoff accepted: no cross-shard / cross-epoch deltas until recompaction — exactly the tradeoff beagle already took by dropping git repack; recovered globally at GIT-006 epoch recompaction.

Blockers and bummers

  • Branch model — RESOLVED, no fork: Store and PackLog both spec one flat object pool per project (branches are REFS rows, no per-branch object dirs), and the impl ALREADY does this — dog/HOME.c HOMEBranchDir discards the branch arg and resolves every consumer (keeper packs/idx, graf idx, refs) to <root>/.be/<project>/, and KEEPCreateBranch is a no-op ("a branch is a REFS row, not a dir — nothing to materialise on disk"). KEEP.h's "Phase 2 multi-branch layout" + PAST/DATA prose is STALE documentation of an abandoned design. So GIT-002 targets the one flat pool unconditionally; OFS bases cannot cross branch dirs because there are none, and the only droppability rule is the existing "no cross-PROJECT bases" (Store §GC). Residue: a doc/code cleanup of the stale KEEP.h comments + any vestigial PAST/DATA scaffolding (which now scans the single shard dir) — file a DIS cross-ref.
  • REF read path cannot be deleted until every store is migrated; GIT-004 lands only after GIT-006 has rewritten on-disk logs (and after any long-lived ~/.be real stores are recompacted).

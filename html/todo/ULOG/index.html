ULOG tickets: append-log & store durability

Discrepancies in dog/ULOG.* — the append-only <ts>\t<verb>\t<uri> event log (worktree .be / store wtlog) and the refs/.refs.idx tip files built on it. Durability and crash-safety of these logs. Each entry is one ticket with its live state; it drops off once the ticket closes. The method is Issues.

Active tickets

  • ULOG-001 — *LANDED 9cfa9639.* Root cause: be amplified a torn ULOG into total loss — a NUL at byte 0 made ulog_scan_log read the log as empty, then ULOGClose→FILETrimBook→ftruncate(0) (abc/FILE.c:945) destroyed the surviving bytes (wtlog+refs are both ULOGs; .keeper packs are off this path, hence objects survived the 2026-06-09 incident). Fix: ulog_scan_log refuses a mid-content NUL with non-NUL bytes ahead (new ULOGTORN) and ULOGOpenBooked restores the original on-disk size on refusal — the torn log is left byte-for-byte for out-of-band repair, never zeroed. Regression dog/test/ULOG.c (torn-first-byte/torn-mid; fail pre-fix, pass post-fix). NB the initial-NUL trigger (ENOSPC / kill-mid-write / clone) was not pinned — H3 concurrent-race did NOT repro, H2 9p ruled out (all ext4). Follow-up candidates: clone of an empty/elided-refs store writes a 0-byte refs + be get HOME-escape; latent O_TRUNC seed at sniff/SUBS.c:410/420.

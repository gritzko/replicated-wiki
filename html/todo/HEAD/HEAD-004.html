HEAD-004: be head be:// fetches subs from .gitmodules URL, not parent

For a be:// (beagle-scheme) remote, be head recursing into a submodule goes STRAIGHT to the sub's declared .gitmodules URL instead of fetching the sub from the SAME parent source first. be head be://localhost/beagle reached into abc and tried https://github.com/gritzko/libabc.git over ssh → github.com:22 refused → WIREUNRCH, aborting the whole peek. A beagle parent keeps every sub in its own shard (Submodules/submodule-parent-source-fetch), so the parent source MUST be the primary candidate; the declared URL is a fallback used only when the parent has no copy. be get already forwards --source (GET-011); HEAD's recursion does not. Relates to HEAD-003.

Input

Context

  • Repro (from ~/todo/DIS-045): be head be://localhost/beagle →

be: head ., be: head abc, then ssh: connect to host github.com port 22: Connection refused, remote https://github.com/gritzko/libabc.git is unreachable, Error: WIREUNRCH, BEDOGEXIT — the peek dies on a sub fetch.

  • The sub abc/libabc lives in the parent shard; be://localhost/beagle

can serve it. Going to github first is both wrong and fatal when offline.

  • be get forwards the in-flight source as --source <src_uri> so sniff's

SubMount builds the PRIMARY candidate from the same remote (beagle/SUBS.c:485-502, GET-011). HEAD's sub recursion omits it.

Goals

  • be head be://<parent> resolves each sub's pin from the PARENT source

first; the .gitmodules URL is tried only if the parent lacks the objects.

  • A dead declared URL (github offline) does NOT fail the peek when the parent

can serve the sub — no WIREUNRCH abort.

Constraints

  • be-scheme parents only; a git-source parent keeps git's own recursion

(the --source flag is omitted when empty — SUBS.c:488).

  • Fallback order is parent-source → declared .gitmodules URL, in that order;

never the reverse (submodule-parent-source-fetch).

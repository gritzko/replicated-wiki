MEM-045: bro — BROOpenFile leaks the booked file map on the two fallible call()s before commit

In the interactive pager, BROOpenFile (BRO.c:900) FILEMapRO's a file into a booked FILE_WANT_BUFS slot and stores it in st->filesidx.mapped, but only registers/commits the slot at the very end — so the two fallible call()s in between early-return and leak the booked map. The path is reached by opening a file (Enter / l / click → BROTryOpen → BROOpenFile) when bro's prior state makes one of those calls fail; the EXEC / MEM-020 tests don't exercise it. From the MEM sweep.

Issues

A booked map is committed late, so intervening error returns leak it.

  • BRO.c:900 BROOpenFile: FILEMapRO into a booked slot at :900, commit at the end; the two call()s between leak the map on error.

Blockers

None; bro-internal, interactive path (not fuzzed).

Planned

Commit the slot before the fallible calls, or unmap on their error returns.

  • Repro first (CLAUDE §17): drive BROOpenFile with a state that fails one of the intervening calls; assert no map leak.
  • Either register the booked map immediately (so BROClose frees it) or FILEUnMap on each intervening error branch.

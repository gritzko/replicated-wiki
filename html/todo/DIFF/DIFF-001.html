DIFF-001: be diff: must recurse into submodules and render gitlink/pin changes

be diff:?<sha> on a commit whose only change is a *submodule pin bump* shows NOTHING. A pin-bump commit's sole change is the sub's gitlink pointer (old-pin → new-pin), which the diff renderer does not emit, and diff: never recurses into the submodule to show what actually changed there — so the real work is invisible from the parent. Repro (in a worktree): be diff:?29df3828 (the ABC-002 pin bump, abc b8ed6277 → d8656f41) → empty output, even though the sub moved by a whole commit. diff: should render the gitlink change AND recurse into the sub to show its commit-range diff, the way GET/POST/PATCH already recurse. See Verbs, Submodules, CLAUDE.

Issues

A gitlink-only commit diffs to nothing, and the sub's content is never reached.

  • be diff:?<sha> on a pure pin-bump commit emits no output — the <sub> gitlink change (old pin → new pin) is not rendered as a diff line.
  • diff: does not recurse into the submodule, so the actual changes between the old and new pins (the sub's commit-range diff) are never shown.
  • Consequence: a parent commit that advances a sub by real work looks empty; review/audit of a pin bump is impossible from the parent.
  • Surfaced verifying ABC-002 (29df3828 bumps abc b8ed6277→d8656f41; the FILEErr change lives in the sub, invisible from be diff:).

Blockers

None. Localized to the diff: renderer + the submodule-recursion the other verbs already do.

Render the gitlink change and recurse into the sub.
